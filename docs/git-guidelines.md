# Git 提交规范

本项目提交信息采用 Conventional Commits 结构，但提交说明使用中文，方便从历史记录里快速看懂改动意图。

## 格式

```text
<type>(<scope>): <中文摘要>

<中文正文，说明为什么改、改了什么、影响范围是什么>
```

示例：

```text
feat(records): 增加记录删除确认流程

新增可复用确认弹窗，用于记录删除等危险操作。

列表页删除成功后只从当前前端数组移除记录，不重新拉取列表。
详情页删除成功后跳转回记录列表。
同步统一列表页和 Dashboard 最近记录的 hover 状态。
```

## 常用 type

- `feat`: 新功能或新增用户可见能力
- `fix`: 修复 bug
- `refactor`: 重构，不改变用户可见行为
- `style`: 样式调整，不改变业务逻辑
- `docs`: 文档调整
- `test`: 测试相关
- `chore`: 构建、依赖、脚本、工具配置等杂项

## 常用 scope

- `records`: 记录列表、详情、新建、编辑、删除
- `dashboard`: Dashboard 页面
- `record-form`: 记录表单
- `tags`: 标签选择、标签接口、标签展示
- `weekly-review`: 周复盘
- `ai`: AI 总结、AI 推荐标签等
- `docs`: 文档

## 中文规则

- 摘要必须用中文描述真实意图，不要只写 “update” 或 “fix bug”。
- 摘要保持简短，建议 20 个中文字符左右。
- 正文用中文说明关键行为变化。
- 如果包含交互变化，正文要写清楚用户看到的变化。
- 如果包含接口或数据变化，正文要写清楚影响范围。
- 不要写空泛描述，例如“优化代码”“调整样式”“修复问题”。
- 不要把多个无关主题塞进同一个 commit。

## 提交前检查

提交前优先运行：

```powershell
pnpm lint
pnpm build
```

如果只改文档，可以不跑 build，但提交正文里应说明只改文档。

## 本机 Git 路径

当前环境中 `git` 不在默认 PATH，Codex 执行 Git 命令时使用：

```text
E:\software\Git\cmd\git.exe
```
